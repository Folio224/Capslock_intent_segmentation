How I did what! Development

First Part of this file will be about Main Model itself.
    At first Glance I thought it was definitely a Clustering (unsupervised) Problem
    But after seeing data and thought about business challenges ( real-time, better readability, "maybe stronger model" also)
    decided to look this problem as a classification and start building propensity model
Propensity model itself is to predict probability of user doing action based on the features you've got

I started simple EDA, cleaning data, finding patterns, sanity checkings and correlation searching
    1. Missingness and type checking
        at first I found that converted wasn't boolean, instead int64, that was unneccessary - so I converted it
        also with .describe() method I found that scroll_depth_pct and time_to_first_scroll_sec had some missing values. 
            for that reason I checked if they had overlap and missing maybe for same reason on same session ID -
            out of 404-300 rows, only 6 was overlapped - meaning weak pattern and kept both with filled median ( median because data clearly was right skewed)
            and also created new column that had info of missingness - XGBoost could use that.
        in EDA I didn't use log transform for right skewed data, because we were training XGBoost, which uses split system and won't be affected.
    2. One-Hot Encoding
        after looking into data Categorical variables were found - which means one-hot encoding was a must
        since session ID was categorical, but is not designed to be used in training, I didn't one-hot encode it, but everything else was applied.
            I used drop_first=True in order to not have "same info duplicate columns"
                So basically if we had landing_page_variant A/B, now we don't have landing_page_variant_A and landing_page_variant_B, but only one landing_page_variant_B and if it's 1 it means B, 0 = A
    3. Sanity check
        Since we know that user can't be converted, if form is not filled, I check if converted = 1 if form_started = 0 and same for interactions
        also after thinking logically and business-wise, it's definitely a data leakage if we use form_started in training
            for two reasons - we can't know user action until he interacts or not, so it's not "before action", also
            it can be seen as cheating since it directly tells model is he likely to convert or not.
    4. Training
        We splitted our data into train/val/test sets - 80-20 in both situation so final numbers are 64% Train, 16% Val, 20% Test
        We are using XGBoost's XGBclassifier with aucpr metric and binary - cross-entropy loss
        since our target is highly imbalanced - nearly 4% converted rate - we can't use
        accuracy or roc-auc score, since they focus on false positives, which doesn't help us here 
        aucpr - scores only positives performance across all threshold, since target is rare - more helpful

            for this problem we also use scale_pos_weight in order to weight our imbalanced 1's.
                scale_pos_weight = (y_train == 0).sum() / (y_train == 1).sum() - weight is equal to division between those two
                This parameter tells the model that "1 is x times more important than 0"
        
        I am also using early stopping technique in order to avoid overfitting - which is a big risk in such small data
                
    5. Evaluation and Results
        roc-auc score - 0.88
        pr-auc - 0.31 -- mentioning the fact that our random is 4% - so we have 7x lift compared to baseline
        Acc - 0.81
        F1 - 0.27 - low because imbalanced data and hard to tell threshold, we have ranking and tiering, not 1/0s
        Shap values shows that sections_viewed,num_clicks and scroll_depth_pct are the three main features that is used by model
        also second graph shows that higher the section view,num_clik and scroll depth - the higher the probabilty of user's 1.
            INTERESTING FACT about A/B testing - here we can see the early stages of A/B testing's output - that when B -> model predicts higher score for 1 - meaning likely to convert
